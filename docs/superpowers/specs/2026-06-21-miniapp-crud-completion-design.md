# Mini App — CRUD Completion (ритуали/задачі, нотатки, грами) — Design Spec

**Дата:** 2026-06-21
**Статус:** Підтверджено користувачем
**Контекст:** Підпроєкт A з двох (B — редизайн RPG-прогресії XP/level/HP, окремий спек).

---

## 0. Проблема

Користувач підняв 3 конкретні дірки в Mini App:

1. Немає кнопки для редагування/видалення ритуалів і задач.
2. Є кнопка «Нотатку» (quick-add → `inbox_items`), але немає способу переглянути додані нотатки — ні у фронтенді, ні в backend API (`POST /api/inbox` є, `GET` нема).
3. У формі додавання їжі (`Today.tsx`) немає поля для грамів, хоча backend (`add_food`, `FoodIn`) це вже підтримує.

При обговоренні пункту 2 виникло питання: чим Notes/Inbox відрізняється від уже існуючого Щоденника (`/journal`, не в Mini App)? Вирішено: **Щоденник = свідома історія**, **Inbox/Нотатки = звалити швидко, розсортувати пізніше** — і розбір (sort) реалізується по-справжньому, а не залишається порожньою обіцянкою як зараз у боті.

## 1. Ритуали та задачі — перейменування + видалення

### Взаємодія

- **Ритуали:** чекбокс (кружечок) залишається toggle-кнопкою, як зараз. Тап на текст назви/решту рядка відкриває bottom-sheet меню.
- **Задачі:** тап будь-де по рядку (крім кнопки «Готово») відкриває те саме меню.
- **Меню (2 пункти):** «Перейменувати» → inline-інпут з поточною назвою, autofocus, Enter/кнопка «Зберегти». «Видалити» → той самий рядок кнопки змінює текст на «Натисни ще раз» на 2-3с (без модального діалогу); повторний тап у цьому вікні підтверджує видалення.

### Backend

`backend/db.py` (нові функції — id-based delete-функції я прибрав раніше цієї сесії як мертвий код; зараз з'являється легітимний викликач, тож повертаю в новому, цілеспрямованому вигляді):

```python
def rename_ritual(ritual_id: str, user_id: str, title: str) -> bool: ...
def delete_ritual_by_id(ritual_id: str, user_id: str) -> bool: ...  # is_active=False, як і delete_ritual_by_title
def rename_task(task_id: str, user_id: str, title: str) -> bool: ...
def delete_task_by_id(task_id: str, user_id: str) -> bool: ...      # .delete(), як і deltask у боті
```

Усі чотири — з `.eq("user_id", user_id)` для перевірки власника, тим самим паттерном, що вже скрізь у `db.py`.

`backend/api_routes.py` (нові роути):

```
PATCH  /api/rituals/{id}   {title}
DELETE /api/rituals/{id}
PATCH  /api/tasks/{id}     {title}
DELETE /api/tasks/{id}
```

### Frontend

`Rituals.tsx` і `Tasks.tsx` отримують спільний bottom-sheet компонент (див. §4) і inline rename-стан, аналогічно існуючим `toggle`/`complete` обробникам у цих файлах.

## 2. Нотатки (Inbox) з реальним розбором

### Навігація

Новий розблокований тайл у `NavGrid.tsx`: **«13 · Нотатки»**.

### Екран `pages/Notes.tsx`

Список необроблених нотаток (`is_handled = false`), найновіші зверху, текст + дата. Той же стиль рядка, що в `Rituals.tsx`/`Tasks.tsx`. Тап на рядок відкриває те саме bottom-sheet меню з 5 пунктами:

| Пункт | Дія |
|---|---|
| → Завдання | миттєво: створює `tasks` з текстом нотатки, позначає обробленою |
| → Щоденник | миттєво: створює `diary_entries` (mood = null) з текстом |
| → Ідея | миттєво: створює `ideas` з текстом |
| → Зустріч | **додатковий крок**: маленька форма «Дата (обов'язково) + Час (за бажанням)» перед створенням `meetings` — на відміну від інших трьох, зустрічі потребують дату, якої в нотатці нема |
| Видалити | той самий двокроковий confirm, що в §1; технічно — просто `clear_inbox_item`, без створення нового об'єкта |

### Backend

`backend/db.py`:

```python
def inbox_to_task(item_id: str, user_id: str) -> dict | None: ...
def inbox_to_diary(item_id: str, user_id: str) -> dict | None: ...
def inbox_to_idea(item_id: str, user_id: str) -> dict | None: ...
def inbox_to_meeting(item_id: str, user_id: str, date: str, time: str | None = None) -> dict | None: ...
```

Кожна: читає текст нотатки (з перевіркою `user_id`), створює відповідний запис, викликає `clear_inbox_item(item_id, user_id)`, повертає створений запис (або `None`, якщо нотатку не знайдено — наприклад, вже обробили з іншого пристрою).

`backend/api_routes.py`:

```
GET    /api/inbox                      (db.get_inbox вже існує, лімітом 10 — той самий ліміт, що в /inbox бота)
POST   /api/inbox/{id}/to-task
POST   /api/inbox/{id}/to-diary
POST   /api/inbox/{id}/to-idea
POST   /api/inbox/{id}/to-meeting      {date, time?}
DELETE /api/inbox/{id}                 (= clear_inbox_item)
```

## 3. Грами в формі додавання їжі

`Today.tsx`, модалка «🍽 Їжа»: новий необов'язковий інпут **«Грами (за бажанням)»** між «Назва» і «Калорії» — той самий порядок, що в боті (`/addfood Назва [грами] ккал`). `handleAddFood` передає значення в уже існуючий `api.addFoodEntry(initData, name, kcal, grams)` — параметр `grams` там вже описаний у типах, просто не використовувався з UI.

---

## 4. Спільні компоненти / наскрізні рішення

- **Bottom-sheet меню** — один компонент (наприклад `ActionSheet`), що приймає список пунктів `{label, onClick, danger?}` і рендериться знизу екрана (схожий стиль на існуючу quick-add модалку в `Today.tsx`). Використовується в `Rituals.tsx`, `Tasks.tsx`, `Notes.tsx` — щоб не плодити 3 копії однієї й тієї ж розмітки.
- **Двокрокове підтвердження видалення** — спільний хук/хелпер (наприклад `useConfirm(action, timeoutMs=2500)`), що тримає стан «чекаю повторного тапу» і автоматично скидає його через таймаут. Один підхід для рядків ритуалів/задач і для нотаток.
- **Помилки** — як і зараз у проєкті: текстовий рядок червоним кольором над дією, без тостів/алертів.
- **Тестування** — без автотестів (конвенція проєкту): `python -m py_compile backend/*.py` для бекенду; ручна перевірка в браузері та в Telegram WebView для фронтенду.
