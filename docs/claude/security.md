# Безопасность

> Назад: [CLAUDE.md](../../CLAUDE.md)
> Горизонт: от текущего MVP до мультиюзера и продажи. Разбито по приоритету: **сейчас**, **до второго юзера**, **перед продажей**.
> Принцип: безопасность данных — не фича на потом, а условие, при котором апку вообще можно дать другому человеку.

---

## 0. Модель угроз (от чего защищаемся)

| Что защищаем | От чего | Насколько критично |
|---|---|---|
| Токен бота, ключи Supabase, AI-ключи | утечка в git / логи / клиент | 🔴 критично всегда |
| Личные данные юзера (вес, питание, дневник, финансы, замеры) | доступ чужого юзера, утечка БД | 🔴 критично с 2-го юзера |
| Биометрия/здоровье (если носимые) | то же + регуляторка (GDPR) | 🔴 перед продажей |
| Вебхук-эндпоинт | поддельные апдейты от чужих | 🟡 базово закрыто |
| Mini App ↔ бэкенд | подмена identity, чужой `user_id` | 🔴 при появлении фронта |

Чувствительность данных в MY-OS высокая: здоровье + финансы + личные записи. Это не «todo-лист» — утечка реально вредит человеку.

## 1. СЕЙЧАС (критично даже на 1 юзере)

### Секреты
- Все ключи (`BOT_TOKEN`, `SUPABASE_SERVICE_ROLE_KEY`, AI-ключи) — **только в Render → Environment**. Никогда в коде/репо.
- `.env` в `.gitignore` — проверить, что реально игнорируется (`git check-ignore .env`).
- В лог не печатать значения ключей, токенов, тела вебхука целиком.
- При утечке ключа — **ротировать немедленно** (в проекте уже был инцидент с утёкшим service_role — урок усвоен).

### service_role
- `service_role` обходит RLS — это **админ-ключ ко всей БД**. Живёт только на бэкенде, никогда не уходит на клиент.
- Сейчас весь бэкенд работает под ним — приемлемо для 1 юзера, но это и есть главный долг к мультиюзеру (см. раздел 2).

### Вебхук
- Проверка заголовка `X-Telegram-Bot-Api-Secret-Token` уже есть — отклоняет апдейты не от Telegram. Не убирать.
- Секрет выводится из токена бота; при ротации токена меняется автоматически.

### Зависимости
- Фиксировать версии в `requirements.txt`, periodically `pip list --outdated`.
- Не тащить лишние пакеты — каждая зависимость это поверхность атаки.

## 2. ДО ВТОРОГО ЮЗЕРА (обязательно перед тем как дать апку кому-то)

> Пока ты один — данные одни. Как только появляется второй человек, отсутствие изоляции = любой может прочитать чужое. Это блокер, не «улучшение».

### Включить реальную изоляцию данных (RLS перестаёт «спать»)
Сейчас RLS-политики созданы, но бэкенд под `service_role` их обходит. Нужно:
1. **Мост `initData → Supabase JWT`** — бэкенд проверяет подпись Telegram initData, выдаёт/использует JWT с `sub = user_id`.
2. Пользовательские запросы идут под этим JWT (через `anon`-ключ), а не под `service_role`.
3. `service_role` оставить ТОЛЬКО для внутренних задач (отчёты, cron) — не для запросов от лица юзера.
4. Проверить, что КАЖДАЯ таблица с `user_id` имеет политику `user_id = auth.uid()`.

### Верификация Telegram initData (когда появится Mini App)
- Mini App присылает `initData` — бэкенд **обязан** проверить HMAC-подпись по токену бота. Без проверки любой может подставить чужой `telegram_id`.
- Проверять `auth_date` (не старше N минут) — защита от replay (переиспользования старого запроса).
- Никогда не доверять `user_id`, пришедшему с клиента напрямую — только извлечённому из проверенного initData.

### Авторизация на уровне запроса
- Каждый эндпоинт, отдающий данные, фильтрует по `user_id` текущего юзера. Даже если RLS включён — дублировать проверку в коде (defense in depth, «защита в несколько слоёв»).
- Никаких эндпоинтов вида «отдай данные по любому id из параметра» без проверки владельца.

## 3. ПЕРЕД ПРОДАЖЕЙ / ПУБЛИЧНЫМ ЗАПУСКОМ

### Юридическое / приватность
- **Политика конфиденциальности** — обязательна, особенно из-за данных здоровья и финансов.
- **GDPR** (если есть юзеры из ЕС, а Польша = ЕС): право на экспорт данных и право на удаление («забыть меня»). Экспорт уже в планах (JSON/CSV) — добавить полное удаление аккаунта.
- Согласие на обработку при онбординге.
- Если подключаешь носимые устройства — биометрия это особо чувствительная категория, отдельное согласие.

### Хранение и доступ
- Медиа (фото еды, замеров, голос) в Supabase Storage — **приватные бакеты**, доступ только владельцу через signed URL, не публичные ссылки.
- Бэкапы БД (Supabase Pro даёт) — проверить, что включены.
- Двойная проверка: ни один публичный бакет, ни одна таблица без RLS не уехала в прод.

### Платёжные данные (если будет оплата)
- НЕ хранить карты у себя. Только через провайдера (Stripe/Paddle/ЮKassa) — они берут PCI-комплаенс на себя.

### Операционка
- Ротация всех ключей перед публичным запуском (финальный аудит из роадмапа, Фаза 5).
- Rate limiting на эндпоинтах — защита от абуза и от выжигания бесплатных AI-лимитов.
- Мониторинг аномалий (резкий рост запросов = кто-то долбит API).
- 2FA на аккаунтах Supabase, Render, @BotFather, GitHub — взлом твоего аккаунта = взлом всего.

### AI-специфика
- Не отправлять в AI-провайдера больше данных, чем нужно для задачи (минимизация).
- Помнить: данные, ушедшие в сторонний AI API, покидают твою инфраструктуру — учесть в политике конфиденциальности.

На 1 юзере НЕ нужно: мост JWT, GDPR-флоу, rate limiting, signed URL. Это всё всплывает на этапе «даю второму человеку» и «продаю». Сейчас закрыто разделом 1 — этого достаточно. Не строить крепость вокруг одиночного бота.
