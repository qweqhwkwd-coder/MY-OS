# MY-OS — Project Memory (CLAUDE.md)

> Личная «ОС для жизни». Telegram-бот + FastAPI + Supabase. Деплой на Render (free).
> **Текущее состояние: bot-first MVP.** Mini App (React) и AI-слой пока не созданы.

Этот файл — точка входа. Детали по доменам — в `docs/claude/` (читай только нужный файл, не всё подряд).

---

## Что это

MY-OS — персональная система трекинга жизни (тренировки, питание, задачи, привычки, финансы, рефлексия) с RPG-слоем: каждое действие даёт XP в одну из 8 характеристик. Цель v1 — один пользователь (автор). Архитектура с RLS заложена под мульти-юзер на будущее.

Полное продуктовое видение (17 модулей) — `MY-OS_product_document.md`.
Рабочий план постройки — `ROADMAP_v2.md`.
Технический срез состояния — `HANDOFF.md`.

## Быстрый старт (команды)

```bash
# Локально (нужен туннель — бот работает только через вебхук, см. architecture.md)
cp .env.example .env          # заполнить значения
pip install -r backend/requirements.txt
cd backend && uvicorn main:app --reload

# Проверка синтаксиса (автотестов нет)
python -m py_compile backend/*.py

# Проверка живости прода
curl https://<service>.onrender.com/health   # → {"status":"ok"}
```

Миграции применяются ВРУЧНУ через Supabase SQL Editor, по одному файлу за раз. См. `docs/claude/database.md`.

## Критичные правила (не нарушать)

1. **`@dp.message()` catch-all (`on_any`) регистрируется ПОСЛЕДНИМ.** Иначе перехватит все команды.
2. **Миграции — по одной, отдельным запуском.** Повторный `create table` падает с `relation already exists`.
3. **Бэкенд ходит под `service_role` — RLS сейчас «спит».** Не полагаться на RLS как на защиту в текущем коде.
4. **Принцип разработки: одна маленькая рабочая функция за раз.** Не строить несколько модулей параллельно. См. `roadmap.md`.
5. **Никаких секретов в репозитории.** `.env` в `.gitignore`. Значения только в Render → Environment. `service_role` никогда не уходит на клиент.
6. **Данные юзера чувствительны** (здоровье, финансы, личное). При любой работе с доступом к данным сверяться с `docs/claude/security.md`. Перед вторым пользователем — обязательна реальная изоляция (RLS + мост initData→JWT).
7. **Коммить после каждого завершённого изменения.** После любой рабочей правки — `git add` + `git commit` с понятным сообщением. Не накапливать несвязанные изменения в одном коммите.
